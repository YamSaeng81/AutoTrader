## 🔴 Tier 1 — 결과 신뢰도를 흔드는 구조적 오류 의혹

### 1. Sharpe / Sortino / Calmar 연환산이 거의 확실히 틀렸다

**증거** — `core-engine/src/main/java/com/cryptoautotrader/core/metrics/MetricsCalculator.java`

```java
private static final BigDecimal ANNUALIZATION_FACTOR = BigDecimal.valueOf(Math.sqrt(365));
...
private static List<BigDecimal> calculateDailyReturns(List<TradeRecord> sellTrades, BigDecimal initialCapital) {
    List<BigDecimal> returns = new ArrayList<>();
    for (TradeRecord t : sellTrades) {
        returns.add(t.getPnl().divide(initialCapital, SCALE, RoundingMode.HALF_UP));
    }
    return returns;
}
```

- 메서드 이름은 `calculateDailyReturns`지만 **실제로는 "거래당 수익률"** 을 반환한다. 3년에 61건 거래면 수익률 시계열 길이는 61.
- 이 시계열로 표준편차를 구하고 `sqrt(365)` 를 곱한다 → **거래 빈도와 실제 일수를 전혀 반영하지 않는다**.
- 3년(1095일)에 61건이면 거래당 평균 간격이 18일이므로, 올바른 annualization factor는 `sqrt(365/18) ≈ 4.5` 수준이어야 한다. 현재 코드는 19.1 을 곱하므로 **Sharpe가 약 4배 과대**.
- Calmar Ratio: `meanTradeReturn × 365 × 100` — 거래당 수익률에 365를 곱해 "연환산"하는 것도 같은 오류. 18일에 한 번 거래하는 전략을 매일 거래하는 것처럼 계산.

**의미**: `PROGRESS.md`에 기록된 "BTC BREAKOUT Sharpe 6.41" 같은 숫자는 **거의 전부 과대평가 되어 있을 가능성이 매우 높다**. 실전 투입 여부 결정 근거 중 하나가 Sharpe 비교였으므로 **전략 선택 판단 자체가 오염**됐을 수 있다.

**해야 할 것**:
- [ ] "거래당 수익률"을 진짜 "일별 equity return"으로 재계산하는 함수 추가 (closed positions 을 일별 리샘플링).
- [ ] 현재 Sharpe/Sortino/Calmar 전면 재계산하여 `PROGRESS.md` 전략 선택 테이블 재검증.
- [ ] 최소한 당장은 `avgReturnPerTrade / stdDevPerTrade` 만 표기하고 "Sharpe-like"로 이름을 바꾸어 해석 착오 방지.

### 2. Walk-Forward 윈도우 겹침으로 OOS 독립성 깨짐

**증거** — `WalkForwardTestRunner.java`

```java
int step = (totalSize - windowSize) / Math.max(windowCount - 1, 1);
```

- 5 window 중 마지막 window 의 start 는 `totalSize - windowSize`, 즉 전체 길이를 5등분이 아닌 **오버랩 슬라이딩**이다.
- 결과: W0 OOS 와 W1 OOS 가 일부 **같은 캔들을 공유**한다. 그런데 `PROGRESS.md` 는 "OOS 합계"로 윈도우 결과를 단순 합산한다.
- **OOS들이 독립 표본이 아니라면 합산 수익률·변동성 모두 통계적으로 오버카운트**된다. SOL에서 "BREAKOUT OOS +12%는 W0 단일 윈도우 +16%에 의존"이라는 분석이 정확히 이 문제를 드러낸다.

**해야 할 것**:
- [ ] Anchored walk-forward 또는 non-overlapping rolling 으로 스위칭 (예: train=2023, test=2024; train=2023~2024, test=2025).
- [ ] 윈도우별 거래 수를 반드시 표기 (거래 0~2건인 윈도우는 통계적으로 의미 없음).
- [ ] overfitting score 대신 Deflated Sharpe / PBO (Probability of Backtest Overfitting) 도입 검토.

### 3. 백테스트 SL/TP 체결 가정이 지나치게 낙관적

**증거** — `BacktestEngine.java` + `ExitRuleChecker.checkCandleExit`

```java
ExitCheck exitCheck = exitChecker.checkCandleExit(nextCandle.getLow(), nextCandle.getHigh(), ...);
// ...
BigDecimal executionPrice = exitCheck.getExitPrice();  // == SL 또는 TP 가격
```

- SL/TP를 H1 캔들의 `low/high` 로 "도달했는지"만 검사하고, **도달 시 정확히 SL/TP 가격에 즉시 체결**된다고 가정.
- 실전에서는:
  1. 지정가 주문이 걸려 있더라도 갭 하락 시 SL 가격보다 훨씬 낮은 가격에 체결될 수 있다 (슬리피지 0% 가정).
  2. SL은 `setScale(8, HALF_DOWN)` 인데 슬리피지 필드가 `ZERO` 로 기록된다 — 그러나 시장가 주문이라면 슬리피지가 붙어야 한다.
  3. 캔들 안에서 SL과 TP가 **동시에 터치**하는 경우, 코드는 "SL 우선"으로 보수적이긴 하지만 실제 path 가 TP 먼저 였을 수도 있다. 확률론적 path 처리 없음.
- H1 캔들에서 `candleLow ≤ SL ≤ candleHigh` 범위가 실제로 중간에 찍힌 후 다시 회복됐을 가능성도 배제됨.

**해야 할 것**:
- [ ] SL/TP 체결에도 슬리피지 % 적용.
- [ ] M15 또는 M5 캔들로 step 을 줄여 intra-H1 path 정확도 향상 (Triple Barrier 방식).
- [ ] SL/TP 동시 터치 시 Monte Carlo로 확률 분포 표본 (혹은 최소한 "양쪽 hit" 경고 로깅).
- [ ] 실전 체결가 vs. 백테스트 가정가 매일 diff 리포트 (매매 후 실시간 비교 루프).

### 4. Partial Fill 이월 로직 — BUY 가 다음 캔들로 이월될 때 position 상태 모순

**증거** — `BacktestEngine.java` 76~97

```java
if (pendingQuantity.compareTo(BigDecimal.ZERO) > 0 && fillSimulator != null) {
    ...
    if (pendingSide == OrderSide.BUY) {
        position = position.add(fillQty);
        ...
    }
}
```

- Partial fill 후 `position > 0` 상태로 다음 iteration 진입 — 이후 SL/TP 검사가 진입가 기준으로 일어나지만 **진입가는 첫 캔들 open 가격, partial 이월분은 다음 캔들 open 가격**이라 average entry price 를 보정하지 않는다.
- 결과: PnL 이 실제와 다르게 계산될 수 있음 (소액 거래에서는 영향이 미미하지만 volume 작은 알트코인/이월이 여러 번 발생하는 종목에서는 문제).
- 또한 pending 이월 중에 전략이 SELL 신호를 내는 경우 pending 을 취소하지만 이미 체결된 부분수량의 entry price 는 갱신되지 않음.

**해야 할 것**:
- [ ] Weighted-average entry price 재계산 로직 추가.
- [ ] Partial Fill 테스트 케이스 보강 (현재 `FillSimulatorTest` 는 함수 단위 테스트만).

### 5. 실전 리스크 체크가 실제 자본이 아닌 `portfolio_limit_krw` 를 쓴다

**증거** — `RiskManagementService.calculateLossPct`

```java
return realizedPnl.abs().divide(portfolioLimit, 4, RoundingMode.HALF_UP).multiply(new BigDecimal("100"));
```

- 일일/주간/월간 손실률 분모가 **`risk_config.portfolio_limit_krw`** (기본 1,000만원 하드코딩).
- 사용자는 소액(1만원 DOGE 세션)부터 시작하는데, 리스크 한도는 1,000만원 대비로 계산되므로 **손실 30만원이 나도 3% 로 보여 서킷 브레이커가 작동하지 않는다**.
- 또한 `realizedPnl ≥ 0` 이면 `ZERO` 를 반환 — 한 구간에 손실·이익이 혼재한 경우 "순이익이라 loss 0%"로 읽혀 실제 변동성을 감춘다.

**해야 할 것**:
- [ ] `portfolioLimit` 을 실제 운영 중인 세션들의 `initialCapital` 합으로 동적 계산.
- [ ] 구간 손실률은 "연속 손실" 또는 "drawdown" 으로 재정의 (이익이 상쇄해도 리스크 없다고 판정하는 건 위험).
- [ ] 세션별 independent circuit breaker 이미 존재하지만 global risk check 와 중복 검사가 어떤 상황에 발동하는지 flow 문서화.

### 6. StrategyLog "4h 적중률 = 전략 성과" 라는 대체재 근사 오류

**증거** — `StrategyWeightOptimizer.computeWeights`

- 가중치 갱신의 기반이 `return4hPct > FEE_THRESHOLD` 인 "신호 적중률".
- 이는 **신호 방향**의 정확성일 뿐, 실제 SL/TP 체결·수수료·슬리피지 등 매매 손익은 고려하지 않는다.
- 신호는 맞았지만 SL에 먼저 걸려 손실 확정된 케이스, 또는 신호는 틀렸지만 TP 트레일링으로 이익 확정된 케이스가 모두 왜곡된다.
- 게다가 `DEFAULTS` 에는 `COMPOSITE_BREAKOUT / COMPOSITE_MOMENTUM` 두 개만 등록 — V1/V2/ICHIMOKU 등 이미 실전 후보로 거론되는 전략들은 자동 최적화 대상 밖.
- **결과**: PROGRESS.md "P1-0 완료" 라고 하지만 옵티마이저가 실제로 전략 선택에 의미 있는 영향을 주지 않을 개연성이 크다.

**해야 할 것**:
- [ ] "종료된 포지션의 realizedPnl / fee 적용 후 수익률" 기반으로 가중치 계산.
- [ ] V1/V2/ICHIMOKU 까지 `DEFAULTS` 에 포함 + 레짐별 기본값을 WFT 결과로 근거 링크.
- [ ] Optimizer 실행 로그 + `WeightOverrideStore` 스냅샷을 DB에 이력으로 저장 (현재 in-memory 로 추정).

## 🟡 Tier 2 — 운영 안전성·동시성 의심

### 7. LiveTradingService 1775줄 = God Class

- `reconcileOnStartup` / `reconcileClosingPositions` / `reconcileOrphanBuyPositions` / `onRealtimePriceEvent` / `executeStrategies` / `executeSessionBuy` / `executeSessionSell` / `finalizeSellPosition` — 모두 한 클래스에 있고 각자 `availableKrw`·`totalAssetKrw`·`positionRepository` 를 수정한다.
- 스케줄러 빈도: `executeStrategies` 60s, `reconcileClosingPositions` 5s, `reconcileOrphanBuyPositions` 30s, WebSocket 이벤트(실시간, 코인당 5s throttle).
- **동시 수정 위험 시나리오**:
  - `reconcileOrphanBuyPositions` 에서 `closeIfOpen` 호출 + `session.availableKrw += invested` 가 수행되는 동안 `executeSessionSell` 에서 매수 취소를 감지하고 **같은 금액을 복원**하려 하면 중복 복원. 현재 `closeIfOpen` atomic 가드가 있지만, 이후 `session` save 는 **read-modify-write** 이라 JPA 낙관적 락 없으면 last-write-wins.
  - `finalizeSellPosition` 의 `session.setTotalAssetKrw(...)` 와 `updateSessionUnrealizedPnl` 의 같은 필드 수정이 겹치면 덮어쓸 수 있음.
- LiveTradingReliabilityTest 1개로는 이 수많은 race를 검증할 수 없다.

**의심해야 할 것**:
- [ ] `LiveTradingSessionEntity` 에 `@Version` (낙관적 락) 붙어 있는지 확인. 없으면 race 상존.
- [ ] Order/Position 쓰기 경로를 수량 단위로 일관되게 (balance 이벤트 스트림 → reducer) 리팩터링 검토.
- [ ] 실전매매 중단 시 재시작 시 "복구 시나리오"별 통합 테스트 작성 (PENDING·SUBMITTED·PARTIAL·FAILED 각 state × OPEN/CLOSING 교차).

### 8. 세션당 `coinPair == coinPair` 암묵 가정

`finalizeSellPosition` 은 "해당 coinPair 에 OPEN 포지션이 남아있는지"로 `totalAssetKrw` 를 결정한다. 세션이 여러 코인을 보유할 가능성을 코드가 막지 않으나(엔티티에는 single coinPair field 만 있지만), 이 동작은 "세션당 1코인"을 가정한다. 다중 세션으로 복수 전략을 한 코인에 돌리는 경우(`createMultipleSessions`) 세션 간 포지션 상태가 독립적인지, 아니면 Upbit 계좌 잔액 공유로 인해 충돌하는지 **명확한 증거가 없다**.

**해야 할 것**:
- [ ] "동일 코인에 다중 세션이 BUY 를 동시에 쏘는 경우" 시나리오 수동 검증.
- [ ] Upbit 계좌 실제 잔액 vs. 모든 세션의 `availableKrw` 합산이 일치하는지 주기적 reconcile 도입 (또는 수동 확인).

### 9. WebSocket 급등락 감지의 단일 장애점

- `upbit-ws-scheduler` 스레드 → `publishEvent` → `@Async("marketDataExecutor")` 로 디커플링.
- WS 연결이 죽어있을 때는 `ExchangeHealthMonitor.setWebSocketConnected(false)` 로 상태만 표시 — WS 끊겨도 5초 throttle 기반 실시간 손절이 "조용히 멈출" 수 있다.
- 현재 코드에는 **WS 끊김 지속 시 fallback 으로 REST 현재가 폴링**이 없는 것으로 보인다. 거래소는 살아있는데 WS만 끊긴 상태가 가장 위험.

**해야 할 것**:
- [ ] WS 연결 손실 지속(>30s) 감지 시 REST ticker 폴링으로 전환 또는 알림 후 자동 SL 보수화.
- [ ] 실시간 SL 마지막 점검 시점을 세션에 기록하고 "X분 미체크 → 경고" 알림.

### 10. 비상 정지 시 "시장가 청산"의 연쇄 충격 미검증

`emergencyStopAll` → 모든 세션 동시 시장가 SELL. 다음 문제가 고려되지 않음:
- 업비트 API rate limit (초당 8건, 분당 200건) 에 걸리면 일부 주문이 실패/재시도 → 주문 누락 가능.
- 다수 세션이 동일 코인에 BUY/SELL 중이면 한 번에 취소·매도 → 시장가 슬리피지 폭증.
- Telegram/Discord 알림 전송이 `emergency` 경로에서 동기적으로 수행되면 알림 지연이 청산 지연을 유발할 수도 있음.

**해야 할 것**:
- [ ] 비상 청산 순서 결정 로직(우선순위) 도입.
- [ ] Upbit rate limit 에 대응하는 token bucket / retry with backoff.
- [ ] 비상 청산 dry-run 모드 추가 (실제 주문 안 쏘고 시나리오만 기록).

### 11. 전략 19종 중 실전 후보는 3개뿐

- 단일 11종 + 복합 8종 = 19종이라고 자랑하지만 실전 매매 대상은 `COMPOSITE_BREAKOUT / COMPOSITE_MOMENTUM / COMPOSITE_MOMENTUM_ICHIMOKU_V2` 3개.

**해야 할 것**:
- [ ] 전략별 운영 가능 여부 매트릭스 (enabled / blocked / experimental / deprecated).
- [ ] 전략 코드는 있지만 실전 투입 경로가 없는 것들은 명시적으로 `@Experimental` 또는 레지스트리에서 분리.

### 12. 파라미터 튜닝 look-ahead 위험

- `MACD (14,22,9) BTC +151.9%` 같은 "파라미터 최적화 단일전략" 결과 — 튜닝 기간·검증 기간 분리 여부가 문서에 없음.
- 2023~2025 3년 데이터로 **파라미터를 고르고** 같은 3년 백테스트를 돌렸다면 전형적 overfitting.
- 단일 MACD 결과(+151.9%)가 COMPOSITE_BREAKOUT +104% 보다 높은데 채택 안 된 이유도 불명.

**해야 할 것**:
- [ ] 파라미터 탐색은 2023~2024 로 제한, 검증은 2025 로. 문서화 필수.
- [ ] 단일 전략 백테스트 매트릭스 재생성 + 기간별 독립 분리.

### 13. 데이터 스냅샷 편향 위험

- 실적합 코인 결과는 전부 **이미 발생한 2023~2025 강세장**. 2022 약세 + 2018 크립토 winter 같은 기간이 없다.
- DOGE V2 +134% 의 대부분은 2024 상승장 특정 구간일 가능성 (월별 리턴 분포 확인 필요).
- 2023-01~2025-12 데이터가 "모든 레짐 포함"이라 문서에 적혀 있지만, 실제로는 강세/횡보/약한조정 중심.

**해야 할 것**:
- [ ] 월별 수익률 분포 표준편차 + skewness 확인 → 한 달에 몰려 있는지 검증.
- [ ] 2022 약세장 데이터 수집 + 재백테스트 (거래소에서 historical candles 내려받을 수 있음).

### 14. 실전/백테스트 일치성 (Paper → Live 갭) 증거 부족

- `BacktestEngine` 과 `LiveTradingService` 가 모두 `ExitRuleChecker` 를 공유하는 것은 좋다.
- 그러나 **백테스트는 SL/TP 정확 체결**, 실전은 **시장가 체결 + 실제 슬리피지** — 이 차이가 얼마나 되는지 측정한 리포트가 없다.
- "실전 체결가 vs. 백테스트 가정가" 의 일일/주간 diff 리포트가 없으면 갭이 벌어져도 탐지 불가.

**해야 할 것**:
- [ ] 실전매매 각 거래마다 "동일 시점·전략 백테스트"를 재현해 diff 저장 (execution vs. simulated).
- [ ] 누적 drift 가 X% 이상이면 알림.

## 🟢 Tier 4 — 테스트·관측·인프라 갭

### 15. 테스트 커버리지

- 전체 Java 파일 190+개 대비 테스트 29개. 아래 영역이 **거의 비어 있다**:
  - `LiveTradingService` (1775줄): 테스트 1개
  - `OrderExecutionEngine` (705줄): 테스트 없음 (찾지 못함)
  - `PaperTradingService` (765줄): 테스트 없음
  - `StrategyWeightOptimizer`: 테스트 없음 → 최근 변경(P1-0) 검증 안 됨
  - `SignalQualityService`: 테스트 없음 → catchup 로직 race 미검증
  - `BacktestJobService`: 테스트 없음
  - `ExitRuleChecker`: 전용 테스트 없음 (가장 중요한 리스크 규칙)
- 전략 테스트는 "단일 캔들 시퀀스 → 예상 시그널" 위주. **멀티 캔들 + 다이버전스 + 상태 누적** 검증은 VolumeDelta 외엔 드물다.
- 통합 테스트(`IntegrationTestBase`) 는 있지만 Upbit 실제 호출 모킹 여부 불분명.

**해야 할 것**:
- [ ] `ExitRuleChecker` 전용 테스트 (SL/TP/트레일링/동시 hit/부분체결 연계).
- [ ] `OrderExecutionEngine` 상태머신 전이 테스트 (PENDING → SUBMITTED → FILLED/PARTIAL/CANCELLED/FAILED 모든 경로).
- [ ] `MetricsCalculator` 수식 검증 테스트 — 위에서 지적한 annualization 오류를 재현해야 함.
- [ ] 백테스트 결정론 테스트: 같은 입력 → 같은 트레이드 시퀀스 (실패 시 비결정적 의존성 존재).

### 16. 관측성(Observability) 공백

### 17. 데이터 백업·복구 드릴

## 📌 결론 — 어디까지 의심해야 하는가

지금까지 축적한 백테스트 결과·전략 선택·Walk-Forward 점수는 **핵심 지표(Sharpe/Calmar/OOS 합계) 계산 방식이 틀렸을 가능성이 매우 높기 때문에**, 대부분의 "근거"가 재검증 대상이다.

특히:

1. **수치 신뢰 등급 A → C 로 강등**: Sharpe 6.41, OOS 합계 같은 값을 근거로 삼는 의사결정은 잠정 보류.
2. **실전매매 금액 증액 보류**: 리스크 체크 분모 오류(Tier 1 §5) 가 해결되기 전까지는 소액 유지.
3. **가장 먼저 고칠 것**: `MetricsCalculator` + `WalkForwardTestRunner` + `RiskManagementService.calculateLossPct`. 이 3개를 수정해야 그 이후 분석이 의미를 가진다.
4. **그 다음**: `LiveTradingService` race 보강 + 테스트 커버리지 + 실전/백테스트 drift 트래커.

완벽하게 작동하는 것처럼 보이는 시스템일수록, 지표 계산 오류나 암묵 가정 한 개가 전체 결론을 뒤집을 수 있다. 이 문서는 **그런 지점들을 나열하는 것이 목적**이며, 여기 적힌 내용 중 일부는 실제로는 문제가 아닐 수도 있으나 **반드시 검증 없이 넘어가지 말 것**.
