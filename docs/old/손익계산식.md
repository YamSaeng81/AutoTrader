📈 자동매매 손익 계산 시스템 스펙 (v1.0 - Refined)1. 핵심 개념 정의 (Core Concepts)1.1 원금 (Principal)정의: 초기 투자 금액 및 이후의 입출금이 반영된 순수 투입 자산.공식: Principal = 초기자산 + 추가입금 - 출금1.2 평가 자산 (Equity)정의: 현재 시점의 총 자산 가치 (현금 잔고 + 보유 코인의 현재 가치).공식: Equity = 현금잔고(Balance) + (보유수량 × 현재가격)1.3 실현 손익 (Realized PnL)정의: 포지션을 청산(매도)하여 지갑에 확정된 실제 수익 또는 손실.공식: Realized PnL = Σ(매도 정산금액 - 매수 원금 - 총 수수료)1.4 미실현 손익 (Unrealized PnL)정의: 현재 보유 중인 포지션을 현재 가격으로 모두 팔았을 때 예상되는 평가 손익.공식: Unrealized PnL = (현재가격 - 평균매수가) × 보유수량1.5 총 손익 (Total PnL) 및 수익률 (Return %)총 손익: Total PnL = Realized PnL + Unrealized PnL수익률: Return Rate (%) = (Total PnL / Principal) × 1002. 거래 단위 계산 로직 (핵심)거래소의 일반적인 수수료 징수 방식(현금(Base)에서 수수료 차감)을 기준으로 보강했습니다.2.1 매수 (BUY)주문금액: 매수수량 × 매수가격수수료: 주문금액 × 수수료율총 차감 현금(Balance): 주문금액 + 수수료👉 평균 매수 단가 업데이트 (가중평균, WAC 방식)Avg Price = (기존 총 매수금액 + 신규 주문금액) / (기존 보유수량 + 신규 매수수량)주의: 수수료는 평균 단가에 포함하지 않고 별도의 비용(Fee)으로 처리해야 수익률 왜곡을 막을 수 있습니다.2.2 매도 (SELL)주문금액: 매도수량 × 매도가격수수료: 주문금액 × 수수료율총 입금 현금(Balance): 주문금액 - 수수료👉 실현 손익 계산 및 누적이번 거래 실현손익 = 주문금액 - (평균매수가 × 매도수량) - 수수료Realized PnL += 이번 거래 실현손익3. 수수료 처리 방식 (Fee Management)작성하신 대로 **방식 1(거래 즉시 차감)**과 **방식 2(별도 누적)**를 모두 가져가는 것이 정석입니다.포지션 단위 누적: 각 포지션(종목)별로 발생한 총 수수료를 Position 테이블에서 추적.계좌 단위 누적: Account 테이블의 total_fee에 모든 거래의 수수료를 합산하여 플랫폼 이용 전체 비용을 파악.4. 상태 관리 구조 (DB 설계 보강)실제 백엔드(Spring Boot, Python 등)와 ORM을 사용하여 Ubuntu 서버의 Docker 환경 등에 DB를 구축할 때 필요한 필수 필드를 추가했습니다.4.1 계좌 (Account)필드명타입설명보강 포인트idPK계좌 고유 IDprincipalDecimal원금balanceDecimal사용 가능한 현금 (예: KRW, USDT)equityDecimal총 평가 자산realized_pnlDecimal총 실현 손익total_feeDecimal누적 총 수수료updated_atTimestamp마지막 자산 업데이트 시간잔고 동기화 체크용versionInteger낙관적 락(Optimistic Lock)동시성 제어용 (필수)4.2 포지션 (Position)필드명타입설명보강 포인트symbolString거래 쌍 (예: BTCUSDT)quantityDecimal현재 보유 수량avg_priceDecimal평균 매입 단가unrealized_pnlDecimal미실현 손익position_feeDecimal해당 종목 누적 수수료개별 종목 수수료 추적용is_activeBoolean포지션 보유 여부전량 매도 시 false (히스토리 보존)4.3 거래 내역 (Trade History)필드명타입설명보강 포인트trade_idPKDB 내부 거래 고유 IDorder_idString거래소 발급 주문 IDAPI 연동 및 대사(Reconciliation)용symbolString거래 쌍typeStringBUY / SELLpriceDecimal실제 체결 가격quantityDecimal실제 체결 수량feeDecimal부과된 수수료realized_pnlDecimal매도 시 발생한 실현 손익매수 시에는 0executed_atTimestamp실제 체결 시간5. 실무 적용 시 반드시 고려해야 할 함정 (Critical Pitfalls)동시성 문제 (Concurrency & Race Condition) ⚠️문제: API에서 체결 웹소켓 데이터가 1밀리초 간격으로 2개 연속 들어올 때, 잔고(balance)가 덮어씌워지는 현상.해결: DB 업데이트 시 버전 관리를 통한 낙관적 락(Optimistic Locking)을 적용하거나, 체결 내역을 메시지 큐에 넣고 순차 처리해야 합니다.부동 소수점 오류와 자투리 코인 (Dust Issue) ⚠️문제: 0.1 + 0.2 = 0.30000000000000004 같은 프로그래밍 언어의 부동 소수점 오차.해결: 금액과 수량은 반드시 Decimal (또는 BigDecimal) 자료형을 사용해야 합니다. 또한 전량 매도를 했는데 소수점 8자리 이하 수량이 남아 포지션이 종료되지 않는(Dust) 현상을 방지하기 위해 잔량 임계치 처리가 필요합니다.슬리피지 (Slippage) 처리시뮬레이션(백테스트)과 실제 서버의 가장 큰 차이입니다. 시장가 주문 시 내가 요청한 가격(price)이 아닌 거래소 호가창에 맞춰진 실제 체결 가격(executed_price)을 기준으로 평균 단가를 계산해야 합니다.평균 단가 오염 방지사용자가 거래소 앱을 켜서 수동으로 코인을 추가 매수/매도하는 경우 봇의 DB와 거래소의 실제 잔고가 틀어집니다. 봇 구동 시 주기적으로 거래소 API를 호출해 실제 Balance와 Position을 덮어쓰는 동기화(Sync) 로직이 필수입니다.
